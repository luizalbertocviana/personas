# Personas

`init-personas.sh` bootstraps a file-based workflow for AI-assisted software development. It generates a shared instruction file, a project state file, working directories for capability tracking, and a set of persona prompts that route one issue at a time through design, implementation, verification, review, documentation, and project-health work.

## What the init script creates

Run:

```bash
./init-personas.sh
```

The script creates these paths if they do not already exist:

- `instructions.md`
- `STATE.md`
- `changes/.gitkeep`
- `specs/.gitkeep`
- `codebase/.gitkeep`
- `.personas/analyst.md`
- `.personas/architect.md`
- `.personas/developer.md`
- `.personas/documentation.md`
- `.personas/investigator.md`
- `.personas/mapper.md`
- `.personas/monitor.md`
- `.personas/refiner.md`
- `.personas/reviewer.md`
- `.personas/security.md`
- `.personas/tester.md`

It is safe to re-run. Existing files are skipped, while `changes/`, `specs/`, `codebase/`, and their `.gitkeep` files are ensured to exist.

After bootstrapping, the next step is to create `specs.md`, then start your agent with:

```text
read instructions.md and follow it
```

## Session flow

Each session generated by `instructions.md` follows this sequence:

1. Run `bd prime`.
2. Read `STATE.md` fully.
3. Run `git status` and `git log --oneline -5`.
4. Check stale in-progress claims with `bd list --status in_progress --json`.
5. If a claim is stale, inspect its last matching commit and unclaim it only when there is no conflicting uncommitted work.
6. Run `bd ready -n 100 --json | jq ...` to choose the next persona and issue.
7. If an issue is selected, read `bd show <id> --json` fully, including prior notes.
8. Load the referenced change file when required.
9. Read the selected persona file fully and follow only that persona.
10. Record notes, update `STATE.md`, commit, and stop.

Issue context loading is strict:

- If the selected issue is `feature`, `bug`, or an untagged `task`, the issue description must contain `Change file: changes/<slug>.md`.
- If that reference exists and the file exists, the persona reads it fully.
- If that reference is missing for those issue types, work stops until a human fixes the issue.
- For other task types such as `ambiguity`, `plan`, `map`, `security`, `review`, `test`, `refine`, `docs`, and `health`, `specs.md` is the fallback context when no change file is referenced.

## Persona selection order

The generated `instructions.md` uses a fixed priority order. The first matching ready issue wins:

| Order | Persona | Trigger |
|---|---|---|
| 1 | Analyst | Ready `task` labeled `ambiguity` |
| 2 | Architect | Ready `task` labeled `plan` |
| 3 | Mapper | Ready `task` labeled `map` |
| 4 | Security | Ready `task` labeled `security` |
| 5 | Reviewer | Ready `task` labeled `review` |
| 6 | Tester | Ready `task` labeled `test` |
| 7 | Investigator | Ready `bug` without `root-cause:` in the description |
| 8 | Developer | Ready `feature`, ready `bug` with `root-cause:`, or ready untagged `task` |
| 9 | Refiner | Ready `task` labeled `refine` |
| 10 | Documentation | Ready `task` labeled `docs` |
| 11 | Monitor | Ready `task` labeled `health` |
| Fallback | Analyst | No ready issue matched, or queue is empty |

## Persona responsibilities

| Persona | Responsibility |
|---|---|
| Analyst | Resolve ambiguities or perform empty-queue gap analysis against `specs.md`, `changes/`, `specs/`, issue history, and the codebase |
| Architect | Write design and verification commands into a change file, then create implementation issues |
| Developer | Implement the selected capability and unit tests, create downstream `refine`, `test`, and optional `docs` issues |
| Tester | Write and run integration/E2E verification, file bugs rather than fixing code |
| Refiner | Make one focused quality improvement and file remaining findings |
| Reviewer | Review a completed capability, file findings, and archive `changes/<slug>.md` to `specs/<slug>.md` only when clean |
| Documentation | Write or update docs based on actual implementation only |
| Security | Audit for vulnerabilities and file security findings |
| Investigator | Diagnose bugs lacking a root cause and create a derived bug with `root-cause:` evidence |
| Mapper | Refresh `codebase/` from verified structural observations and absorbed `STATE.md` notes |
| Monitor | Produce a health report and file issues for drift, stalls, hotspots, or test-ratio problems |

## Change-file workflow

The generated personas use `changes/<slug>.md` as the in-flight specification for a capability. The standard template includes:

- `## Why`
- `## Preferences`
- `## Scope`
- `## Out of scope`
- `## Constraints`
- `## Decision log`
- `## Open questions`
- `## Design`
- `## Verification commands`
- `## As built`

Typical lifecycle:

1. Analyst creates a change file for a newly identified capability.
2. Architect fills in `## Design` and `## Verification commands`.
3. Developer implements against that change file.
4. Tester, Refiner, Security, and Reviewer use the same change file as shared context.
5. Reviewer moves it from `changes/` to `specs/` when the capability is ready to archive.

## Commit convention

All generated personas use this commit format:

```text
<type>(<scope>): <short description>
```

Valid types in the generated instructions are:

- `feat`
- `fix`
- `refine`
- `test`
- `review`
- `docs`
- `security`
- `investigate`
- `plan`
- `map`
- `monitor`
- `chore`

The scope is normally the change-file slug, or a persona-specific scope such as `analyst`, `mapper`, or `monitor`.

## State and codebase files

`STATE.md` is the shared session memory. The generated file contains these sections:

- `## Current blockers`
- `## Key architectural decisions`
- `## Capability status`
- `## Known concerns`
- `## Session log`

`codebase/` is reserved for verified structural documentation. The generated personas expect it to hold long-lived architecture notes, while `STATE.md` stays the running log.

## Prerequisites

- `bash`
- `git`
- `jq`
- `beads` (`bd`)
- An agent that can read files and run shell commands
